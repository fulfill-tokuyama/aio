# コンテキストフレッシュ付きレビューサイクル — 手順型スキル

このスキルは、UIの実装後に品質を担保するためのレビューループを定義する。同一レビュアーで繰り返すとスコアが甘くなるバイアスを排除するため、改善のたびにレビュアーを kill → spawn し、まっさらなコンテキストで毎回評価する。

## インプット
- 実装が完了した画面名（specs/flow.md のIDで指定）
- 実装内容の概要（ユーザーストーリーIDで指定）

## アウトプット
- UXレビュースコアシート（ux-checklist.md 準拠）
- 合格判定（90点以上）または改善指示リスト
- 実装画面のスクリーンショット（tickets/ に保存）

---

## レビューサイクル全体フロー

```
Step 1: 実装完了宣言
    │
    ▼
Step 2: スクリーンショット撮影（自動）
    │
    ▼
Step 3: レビュアーエージェント spawn（新規生成）
    │
    ▼
Step 4: UX評価（ux-checklist.md で採点）
    │
    ├─ 90点以上 ──▶ Step 7: 合格・マージ準備
    │
    └─ 89点以下 ──▶ Step 5: 改善指示
                        │
                        ▼
                    Step 6: 改善実装
                        │
                        ▼
                    レビュアー kill（停止）
                        │
                        ▼
                    Step 3 に戻る（新しいレビュアーを spawn）
```

**最大ループ回数: 5回**。5回で90点に達しない場合は人間（TAKASHI）にエスカレーションする。

---

## 各ステップの詳細

### Step 1: 実装完了宣言

**担当**: 実装エージェント（FE or BE）

実装が完了したら、以下の情報を出力する。

```
## 実装完了レポート
- 画面ID: [flow.md のID]（例: DASH-01）
- ユーザーストーリー: [user-stories.md のID]（例: US-PIPE-01）
- 変更内容: [何を実装/変更したか、箇条書きで]
- 影響範囲: [他の画面への影響があればここに記載]
- 自己チェック:
  - [ ] TypeScript の型エラーなし
  - [ ] ESLint エラーなし
  - [ ] デザイントークン（design-tokens.md）を目視確認
  - [ ] 日本語テキストは sme-copywriting.md を確認
  - [ ] ブラウザでの動作確認済み
```

自己チェックが全て通っていない場合、レビューに進まず先に修正する。

---

### Step 2: スクリーンショット撮影

**担当**: 実装エージェント（自動実行）
**ツール**: Playwright CLI

以下のコマンドでスクリーンショットを撮影する。

```bash
# デスクトップ表示（1280x800）
npx playwright screenshot \
  --browser chromium \
  --viewport-size "1280,800" \
  http://localhost:3000/[対象パス] \
  tickets/[画面ID]-desktop-[日付].png

# モバイル表示（375x812 = iPhone SE相当）
npx playwright screenshot \
  --browser chromium \
  --viewport-size "375,812" \
  http://localhost:3000/[対象パス] \
  tickets/[画面ID]-mobile-[日付].png
```

**撮影ルール**:
- デスクトップとモバイルの2枚を必ず撮影する
- ページ全体をキャプチャする（full page screenshot）
- ログインが必要な画面は、先にログイン処理を挟む
- データが空の状態（Empty State）と、データがある状態の両方を撮る（計4枚）
- ファイル名: `[画面ID]-[desktop/mobile]-[YYYYMMDD].png`

---

### Step 3: レビュアーエージェント spawn

**担当**: オーケストレーター（またはCLAUDE.md の指示に従い自律実行）

**重要: 新しいレビュアーをまっさらなコンテキストで生成する。**

レビュアーエージェントには以下のスキルをプリロードする。
- ux-checklist.md（評価基準）
- design-tokens.md（デザイントークン）
- sme-copywriting.md（テキスト品質基準）

レビュアーに渡す情報:
```
## レビュー依頼

あなたは AIO Insight（FormPilot）の UX レビュアーです。
以下の画面を ux-checklist.md に基づいて評価してください。

- 画面ID: [ID]
- スクリーンショット: [ファイルパス]
- 実装内容: [概要]
- 該当ユーザーストーリー: [user-stories.md のID]

重要:
- あなたはこの画面を初めて見ます。先入観なしで評価してください
- 全20項目を0-5点で採点してください
- 各項目に具体的なコメント（良い点・問題点）を必ず記載してください
- 合計90点以上で合格です
```

**絶対にやってはいけないこと**:
- 前回のレビュー結果をレビュアーに渡さない（バイアスの原因）
- 「前回から改善されているはず」という前提を持たせない
- 実装エージェントにレビューを兼任させない（自分の実装は甘く評価する）

---

### Step 4: UX評価

**担当**: レビュアーエージェント（Step 3 で spawn したもの）

ux-checklist.md のスコアシートテンプレートに従って全20項目を採点する。

**採点時の追加ルール**:

1. スクリーンショットを見て、第一印象を最初にメモする（3秒ルール）
   - 「最初に目に入ったものは何か」
   - 「次にやるべきことが分かるか」
   - 「FormPilotの既存画面と同じサービスに見えるか」

2. ユーザーストーリーの受け入れ条件を1つずつ照合する
   - 条件を満たしていない場合、該当する評価項目のスコアを下げる

3. テキストチェック
   - sme-copywriting.md の品質チェックリストを適用する
   - 英語が残っていないか
   - 専門用語に補足があるか

4. デザイントークンチェック
   - design-tokens.md のカラー・余白・角丸・フォントサイズとの照合

**出力フォーマット**:

```
# UXレビュー: [画面名] ([日付])
## レビュアー: [spawn ID]（レビューループ [N]回目）

## 第一印象（3秒ルール）
- 最初に目に入ったもの: [記載]
- 次のアクションの明確さ: [記載]
- ブランド適合の印象: [記載]

## スコアシート
[ux-checklist.md のテンプレートに記入]

## 合計: [X] / 100点
## 判定: [合格 / 不合格]

## サマリー
### 良い点（Keep）
- [具体的に記載]

### 改善が必要な点（Problem）
- [具体的に記載。優先順位付き]

### 改善提案（Try）
- [具体的な改善方法を記載]
```

---

### Step 5: 改善指示（不合格の場合）

**担当**: レビュアーエージェント → 実装エージェントに伝達

レビュー結果のうち、スコアが3点以下の項目を改善対象として抽出する。

**改善指示のフォーマット**:
```
## 改善指示（レビューループ [N]回目）

### 最優先（スコア0-1の項目）
1. [項目名]: [具体的な改善内容]

### 高優先（スコア2の項目）
1. [項目名]: [具体的な改善内容]

### 中優先（スコア3の項目）
1. [項目名]: [具体的な改善内容]

### 改善しなくてよい項目
- スコア4-5の項目は現状維持
```

**重要**: 改善指示は具体的に書く。「デザインを改善して」ではなく「カード背景色を #111118 に変更し、角丸を rounded-lg（8px）にする」のレベルで記載する。

---

### Step 6: 改善実装

**担当**: 実装エージェント

改善指示に従って修正を実施する。

修正完了後:
1. Step 1 の自己チェックを再実行する
2. Step 2 でスクリーンショットを再撮影する
3. **レビュアーを kill する**（ここが最重要）

---

### Step 7: 合格・マージ準備（90点以上の場合）

**担当**: 実装エージェント + オーケストレーター

1. レビュースコアシートを `tickets/[画面ID]-review-[日付].md` に保存する
2. スクリーンショットが `tickets/` に保存されていることを確認する
3. Git コミットメッセージに以下を含める:
   ```
   feat: [画面名] 実装完了
   
   - ユーザーストーリー: [US-ID]
   - UXスコア: [X]/100（レビューループ [N]回）
   - レビュー詳細: tickets/[画面ID]-review-[日付].md
   ```
4. deploy-flow.md に従ってデプロイ準備に進む

---

## レビューループの監視指標

| 指標 | 健全な範囲 | 注意が必要な状態 |
|------|-----------|----------------|
| 平均ループ回数 | 1-2回 | 3回以上が続く場合、スキルの不備を疑う |
| 初回スコア平均 | 75-85点 | 60点以下が続く場合、実装品質の問題 |
| 最も低い項目 | ばらつきあり | 同じ項目が繰り返し低い場合、スキルに該当ルールを追加 |
| ループ5回到達率 | 0% | 1回でもあればエスカレーション |

---

## エスカレーション基準

以下の場合、TAKASHI に人間レビューを依頼する。

- 5回ループしても90点に達しない
- レビュアーと実装エージェントの間で矛盾した指摘が繰り返される
- 評価項目の解釈で判断が分かれる（例: 「安心感」の主観的評価）
- 新しいデザインパターンで design-tokens.md にルールがない

エスカレーション時に渡す情報:
- 全ループのスコアシート
- デスクトップ + モバイルのスクリーンショット
- 改善が収束しない項目のハイライト
