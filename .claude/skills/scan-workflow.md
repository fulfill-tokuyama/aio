# スキャン→スコアリング→メール送信 パイプライン — 手順型スキル

このスキルは、FormPilot の自動営業パイプラインの全工程を定義する。パイプラインの改修・拡張・デバッグを行うエージェントは、このスキルを読んで全体像を把握した上で作業すること。

## インプット
- スキャン対象の業種・地域の設定（SET-01 スキャン設定画面で管理）
- 実行トリガー（手動実行 or 定期スケジュール）

## アウトプット
- 新規リードの Supabase 登録
- スコアリング済みリードのパイプライン配置
- 営業メールの自動送信
- アクティビティログの記録

## 各ステップの詳細

### Step 1: ターゲット企業リスト生成

**担当**: スキャンエンジン（バックエンド）
**トリガー**: 手動実行ボタン or スケジュール（SET-01 で設定）

**処理内容**:
1. SET-01 のスキャン設定から対象業種・地域を取得する
2. 以下のソースから企業リストを生成する
   - Google検索（業種 + 地域キーワード）
   - 業種別ポータルサイト
   - iタウンページ等の企業データベース
3. 既に pipeline_leads テーブルに登録済みの URL は重複チェックで除外する
4. 1回のスキャンで最大50企業を取得する（API制限考慮）

**出力**: 企業名、URL、業種、地域のリスト

**エラーハンドリング**:
- ソースサイトへのアクセスエラー → 3回リトライ後にスキップ、ログ記録
- 重複検出 → pipeline_leads の url カラムで一意性チェック

---

### Step 2: LLMO対策状況スキャン

**担当**: スキャンエンジン（バックエンド）
**依存**: Step 1 の企業リスト

**処理内容**:
1. 各企業のWebサイトにアクセスする
2. 以下の項目を自動チェックする

| チェック項目 | 判定方法 | 配点 |
|-------------|---------|------|
| 構造化データ | HTML内のJSON-LD / microdata の有無をパース | 20点 |
| E-E-A-T シグナル | 会社概要・代表者情報・実績ページの有無 | 20点 |
| FAQ コンテンツ | /faq, /qa, よくある質問ページの存在チェック | 15点 |
| コンテンツ専門性 | ページ数、テキスト量、独自コンテンツの判定 | 15点 |
| 技術的SEO | HTTPS、モバイル対応、サイトスピード（Lighthouse API） | 10点 |
| 外部言及 | 被リンク数の概算チェック | 10点 |
| AI検索言及テスト | Gemini API で業種キーワード検索、企業名の言及有無 | 10点 |

3. 各項目のスコアを算出する（AIO-knowledge.md の診断ロジック参照）

**レート制限**:
- 1企業のスキャンに最低2秒のインターバルを設ける
- Gemini API は 1分間に最大10リクエスト
- Lighthouse API は 1分間に最大5リクエスト

**エラーハンドリング**:
- サイトアクセス不可（404/500）→ スコア0点で登録、ステータスを「アクセス不可」に
- Gemini API エラー → AI検索言及テストの項目を「未評価」としてスキップ（他項目で採点）

---

### Step 3: AIスコアリング（Gemini API）

**担当**: スキャンエンジン + Gemini API
**依存**: Step 2 のスキャン結果

**処理内容**:
1. Step 2 の各項目スコアを合算して総合スコア（0-100点）を算出する
2. スコアに基づいてランクを付与する（AIO-knowledge.md 参照）

| スコア | ランク | パイプライン優先度 |
|--------|--------|--------------------|
| 0-19 | E（未対策） | 最高（即座にキュー投入） |
| 20-39 | D（危険） | 高 |
| 40-59 | C（要改善） | 中 |
| 60-79 | B（良好） | 低（フォローアップ対象） |
| 80-100 | A（優良） | 対象外（スキャン結果のみ記録） |

3. 営業優先度スコアを別途算出する（業種×企業規模×対策状況×地域）

**Gemini API プロンプト設計のルール**:
- プロンプトは日本語で記述する
- 出力フォーマットは JSON で指定する
- 温度パラメータは 0.2（一貫性重視）
- モデルは gemini-2.0-flash を使用する

---

### Step 4: Supabase へリード登録

**担当**: バックエンド
**依存**: Step 3 のスコアリング結果

**処理内容**:
1. 以下のカラムで `pipeline_leads` テーブルに INSERT する（supabase-schema.md 参照）

```sql
INSERT INTO pipeline_leads (
  company, url, industry, region,
  llmo_score, ai_score, weaknesses, phase
) VALUES (
  '企業名', 'https://...', '業種', '地域',
  75, 82, '[{"category":"...","severity":"...","description":"..."}]', 'discovered'
);
```

2. phase を `'discovered'` に設定する
3. pipeline_activity_log に「スキャンにより新規登録」を記録する

---

### Step 5: フォーム自動探索

**担当**: スキャンエンジン
**依存**: Step 4 で登録されたリード

**処理内容**:
1. 企業Webサイト内で問い合わせフォームを探索する
   - /contact, /inquiry, /form 等のパスを優先チェック
   - ページ内のリンクテキスト「お問い合わせ」「ご相談」を検索
   - メールアドレス（mailto: リンク）を検出
2. 発見した連絡先情報を pipeline_leads テーブルに UPDATE する
3. phase を更新する
   - フォーム発見 → 「フォーム発見」
   - メールアドレスのみ発見 → 「メール発見」
   - 連絡先なし → 「連絡先未発見」（パイプラインから一時除外）

**重要**: 企業のフォームに自動送信は行わない。フォームURLの発見のみ。実際のアプローチはメール送信で行う。

---

### Step 6: メールキュー投入

**担当**: バックエンド
**依存**: Step 5 で連絡先が判明したリード

**処理内容**:
1. 以下の条件を満たすリードを pipeline_leads から抽出してメールキューに投入する
   - llmo_score が 0-59
   - contact_email が NOT NULL
   - pipeline_automation_config の automation_enabled が true
   - 過去30日以内に sent_at が NULL
2. 営業優先度スコアでソートする（高い順）
3. 1日の送信上限を確認する（SendGrid無料枠: 100通/日）
4. pipeline_leads の phase を 'queued' に UPDATE する

**キューの優先ルール**:
1. ランク E（未対策）の高優先業種 → 最優先
2. ランク D（危険）の高優先業種 → 次点
3. ランク E の一般業種
4. ランク C の高優先業種
5. その他

---

### Step 7: 営業メール生成・送信

**担当**: メール自動化エージェント（バックエンド）
**依存**: Step 6 のメールキュー

**処理内容**:
1. email-templates.md のテンプレートに基づいてメール文面を生成する
2. 以下の変数を埋め込む
   - [企業名]: leads.companyName
   - [業種]: leads.industry
   - [担当者名]: leads.contactName（不明の場合は「ご担当者」）
   - [診断URL]: LP の無料診断フォームURL + UTMパラメータ
3. A/Bテスト設定がある場合は、テストグループに応じたバリアントを選択する
4. SendGrid / Resend API でメールを送信する
5. 送信結果を Supabase に記録する
   - email_logs に送信記録を INSERT
   - pipeline_leads の phase を 'sent' に UPDATE、sent_at を設定
   - pipeline_activity_log に記録

**送信タイミング**:
- 平日（月-金）の 10:00-11:00 または 14:00-15:00（開封率が高い時間帯）
- 土日祝は送信しない
- 1企業あたり最低3日のインターバルを空ける

**エラーハンドリング**:
- 送信失敗（バウンス）→ ステージを「送信エラー」に、3回失敗で「無効」に
- レート制限 → キューに戻して次のバッチで再送

---

### Step 8: 反応トラッキング

**担当**: バックエンド（Webhook / 定期チェック）
**依存**: Step 7 で送信されたメール

**処理内容**:
1. SendGrid / Resend の Webhook でイベントを受信する

| イベント | ステージ更新 | アクション |
|---------|-------------|-----------|
| 開封（open） | 「開封」 | アクティビティログに記録 |
| リンククリック（click） | 「興味あり」 | アクティビティログに記録 + 優先度UP |
| 返信（reply） | 「商談」 | Slack通知 + 手動対応フラグ |
| バウンス（bounce） | 「送信エラー」 | メールアドレスを無効マーク |
| 配信停止（unsubscribe） | 「配信停止」 | 以後のメール送信を完全停止 |

2. 各イベントの発生時刻をアクティビティログに記録する
3. 「興味あり」のリードは salesPriority を +20 加算する

---

### Step 9: フォローアップ判定

**担当**: バックエンド（定期バッチ / 毎日1回）
**依存**: Step 8 のトラッキング結果

**処理内容**:
1. 送信後3日経過かつ未開封のリードを抽出する
2. フォローアップ対象かどうかを判定する

| 条件 | アクション |
|------|-----------|
| 未開封 × 3日経過 × フォローアップ0回 | フォローアップメール送信（email-templates.md のテンプレート2） |
| 未開封 × 7日経過 × フォローアップ1回 | 2回目フォローアップ（別の件名で再送） |
| 未開封 × 14日経過 × フォローアップ2回 | 「休眠」に移行。3ヶ月後に再スキャン対象 |
| 開封 × 未クリック × 5日経過 | 追加情報メール送信（業種固有のデータ付き） |
| 「興味あり」× 3日経過 × 未返信 | 商談誘導メール送信（email-templates.md のテンプレート4） |

3. フォローアップメールの送信は Step 7 と同じフローに乗せる

## パイプラインステージ一覧

```
発見 → フォーム探索 → フォーム発見 → キュー → 送信済
  │         │                                    │
  │         └→ 連絡先未発見（一時除外）            ├→ 開封 → 興味あり → 商談 → 受注
  │                                               │                        └→ 失注
  │                                               └→ 未開封 → フォローアップ → 休眠
  └→ アクセス不可（除外）
```

## 監視・アラートルール

| 監視項目 | 閾値 | アクション |
|---------|------|-----------|
| スキャン成功率 | < 80% | エラーログを確認、ソースサイトの変更を疑う |
| メール送信成功率 | < 95% | SendGrid/Resend のステータスを確認 |
| 開封率 | < 15%（3日間平均） | 件名のA/Bテストを優先実施 |
| バウンス率 | > 5% | メールアドレス取得ロジックを見直し |
| 日次送信数 | > 90通 | 送信上限（100通）に接近、プランアップグレードを検討 |
| Gemini API エラー率 | > 10% | プロンプト・モデル設定を確認 |

---

## このスキルの更新ルール

パイプラインに新しいステップを追加・変更する場合は、必ずこのスキルを先に更新してから実装に着手すること。スキルと実装の乖離が発生すると、他のエージェントが古い情報で作業してしまうリスクがある。
